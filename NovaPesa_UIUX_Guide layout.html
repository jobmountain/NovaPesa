SPECS · 03
Modular Architecture Requirement MANDATORY
Show this to your technical co-founder. Non-negotiable.
MANDATORY: NovaPesa MUST use modular architecture where each pillar is developed as an independent module with its own services and database models. This is not optional. It determines whether NovaPesa can scale globally or stays small.
LayerWhat It ContainsWhy It Matters
Core ServicesAuth · User Profile · NovaScore Engine · Notification Service · Payment GatewayShared across ALL 16 pillars. Built first in Sprint 1. Every module depends on these — they must be rock solid.
Pillar ModulesEach pillar = /modules/pillar-name/ with own routes.js · services.js · models.js · controllers.jsIndependent development. Team member per module. Zero cross-module database access — only via Core Services API.
FrontendReact.js. Each pillar = /src/features/pillar-name/ with own pages, components, hooks, api foldersSelf-contained feature folders. Shared UI components in /src/components. Allows parallel frontend + backend development.
NovaScore EngineDedicated service. Receives events from all modules. Recalculates score. Emits score-update events.Central brain. Every action on every pillar can affect the score. Must be event-driven, never tightly coupled.
Module CommunicationModules communicate via event bus (Redis pub/sub) or REST calls to Core Services ONLYIf Trading module goes down, Banking still works. If AI module is slow, community still works. Resilience at scale.
Tell your co-founder: "Each of our 16 pillars must be its own independent module with its own services and database models. They communicate through a shared API layer only — not directly. If they push back and say this is overkill for V1, that is a red flag."
